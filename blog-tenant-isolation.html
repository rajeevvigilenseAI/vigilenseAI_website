How We Handle Tenant Isolation When Every Customer Brings Their Own Database

Nov 14, 2025 · 8 min read · By Raj Choudhary

Multi-tenancy is hard. Multi-tenancy where every customer brings their own database, their own schema, their own access patterns, and their own compliance requirements? That is a fundamentally different engineering problem. At Vigilense, we built our isolation model from the ground up for BYODb, and it goes far beyond logical separation. In the cloud, each tenant's data, compute, and network are fully separated at the infrastructure level.

This post walks through the six isolation layers that protect every tenant on our platform, the cryptographic architecture that underpins them, and the testing regimen that keeps us honest.

Why BYODb Makes Isolation Harder

Most multi-tenant platforms control the data layer. They decide the schema, the storage engine, and the access patterns. That simplifies isolation because you own the entire stack.

BYODb removes that luxury. Every customer has a different database engine, a different schema, a different set of credentials, and a different network topology. You cannot rely on a single shared database with row-level policies and call it a day. You need credential vaults that never co-mingle secrets. You need query sandboxing that prevents a detection rule from one tenant from reaching another tenant's data. You need network segmentation that treats every customer connection as its own trust boundary.

That is the starting point, not the finish line.

The Six Isolation Layers

Our isolation model is built in depth. Compromise of any single layer does not expose tenant data because the remaining five layers continue to enforce boundaries independently.

1. Identity Layer

Every request that enters the system is authenticated with tenant context before it touches any business logic. There is no anonymous path through the platform. Authentication verifies both the user identity and the tenant they belong to, and that pairing is immutable for the lifetime of the session.

We enforce MFA on all accounts. SSO is supported through SAML 2.0 and OIDC, so customers can bring their own identity providers. Sessions are managed with short-lived tokens, and device trust validation ensures that only approved endpoints can establish sessions. If a session token is replayed from an untrusted device, it is rejected.

2. Application Layer

Once a request is authenticated, the tenant ID is injected into every downstream operation. This is not optional. Our application framework enforces it at the middleware level. Every database query, every API call, every log write, every cache lookup carries a tenant identifier. If a code path attempts to execute without tenant context, the request fails immediately.

This design prevents an entire class of bugs where a developer forgets to scope a query. The framework makes unscoped queries impossible.

3. Data Layer

We support three levels of database isolation, and customers choose based on their compliance and security requirements:

  • Row-Level Isolation (Standard): Shared tables with enforced row-level security (RLS) policies. Every query is automatically scoped to the tenant's rows. This is the default for most customers and provides strong isolation with efficient resource utilization.
  • Schema-Level Isolation (Enhanced): Each tenant gets a dedicated schema within a shared database instance. Queries are restricted to the tenant's schema at the database engine level. Cross-schema access requires explicit privilege grants that are never configured in production.
  • Database-Level Isolation (Maximum): Each tenant gets a fully dedicated database instance. No shared storage, no shared processes, no shared connection pools. This is for customers with the strictest regulatory requirements: financial services, defense contractors, and healthcare organizations handling PHI at scale.

4. Encryption Layer

Every piece of data on the platform is encrypted, and every tenant has their own encryption key hierarchy. The architecture works like this:

  • HSM-Protected Master Key: The root of the key hierarchy lives in a hardware security module. It never leaves the HSM boundary.
  • Tenant Key Encryption Key (KEK): Each tenant gets a unique KEK derived under the master key. This KEK wraps all of the tenant's data encryption keys.
  • Data Encryption Keys (DEKs): The actual keys used to encrypt data. These are rotated regularly and are unique per tenant. Each DEK is wrapped by the tenant's KEK, so a compromised DEK from one tenant has zero impact on any other tenant.

All data types are encrypted with AES-256-GCM using per-tenant DEKs. This includes configuration data, database credentials, API keys, alert data, and audit logs. There are no exceptions.

If a tenant leaves the platform, we support crypto-shredding: destroying the tenant's KEK renders all of their encrypted data permanently unrecoverable without touching the underlying storage. No residual data, no recovery window, no ambiguity.

5. Network Layer

Every tenant's traffic is isolated at the network level. This is not just firewall rules on a shared subnet. It is full VPC isolation with strict security groups, private subnets, and a web application firewall (WAF) on all entry points.

Network security groups are configured to prevent any cross-tenant network access. Tenant A's compute cannot open a connection to Tenant B's data store, even if an attacker gains code execution inside a container.

For customer connectivity, we support multiple options depending on security requirements:

  • TLS 1.3: Standard encrypted transport for all API and UI traffic.
  • VPC Peering: Direct network peering between the customer's VPC and their isolated Vigilense environment.
  • PrivateLink: AWS PrivateLink endpoints for customers who need connectivity without traversing the public internet.
  • VPN: Site-to-site VPN tunnels for customers with on-premises infrastructure that needs to reach their Vigilense deployment.

6. Compute Layer

Processing isolation is the layer most platforms skip, and it is the one that matters most when an attacker achieves code execution. Our approach: container isolation per tenant.

Each tenant's workloads run in dedicated containers with enforced CPU and memory limits. Kubernetes namespaces provide the orchestration boundary, and network policies within the cluster prevent cross-namespace traffic. There is no shared memory between tenant containers. A noisy-neighbor problem in one tenant's detection pipeline cannot degrade another tenant's performance, and a container escape in one namespace cannot reach another tenant's workloads.

Full Cloud Tenant Separation

We want to be explicit about this: in the Vigilense cloud deployment, tenant separation is not just a logical construct enforced by application code. It is infrastructure-level isolation.

Each tenant's data lives in isolated storage. Each tenant's compute runs in isolated containers within isolated namespaces. Each tenant's network traffic flows through isolated network segments with independent security group policies. The encryption keys are unique per tenant and managed in dedicated key hierarchies.

This means a compromise of one tenant's environment does not give an attacker a foothold into any other tenant's environment. The blast radius is contained by design, not by hope.

Access Control

Strong isolation is meaningless if access control is weak. Our access model enforces the principle of least privilege at every level:

  • MFA Required: Every account, no exceptions. Hardware keys supported.
  • SSO Integration: SAML 2.0 and OIDC so customers control their own identity lifecycle.
  • Session Management: Short-lived tokens, automatic expiration, re-authentication on sensitive operations.
  • Device Trust: Only registered and compliant devices can establish sessions.
  • RBAC: Role-based access control with granular permissions. Analysts see alerts. Admins configure integrations. No role has more access than it needs.

The BYODb Isolation Challenge: Credentials, Queries, and Segmentation

BYODb introduces isolation challenges that do not exist in traditional SaaS architectures. Every customer's database credentials need to be stored, rotated, and accessed without any possibility of cross-tenant leakage. Every query our detection engine executes needs to be sandboxed so it only touches the intended tenant's data. And every network connection to a customer database needs to be segmented so traffic from Tenant A's queries never traverses Tenant B's network path.

We handle this with dedicated credential vaults per tenant, query execution sandboxes that enforce tenant scope at the engine level, and network segmentation that creates independent connectivity paths to each customer's database infrastructure. The credential vault for each tenant is encrypted with that tenant's KEK, so even our own operators cannot view raw credentials without the key hierarchy producing the correct decryption chain.

How We Test All of This

Architecture diagrams are promises. Testing is proof. Our security validation program includes:

  • Annual Third-Party Penetration Tests: We engage external security firms to conduct targeted penetration tests specifically focused on tenant isolation. The scope includes attempts to access one tenant's data from another tenant's context, privilege escalation across tenant boundaries, and credential extraction from the vault layer.
  • CI/CD Tenant Boundary Tests: Every code change runs through automated tests that validate tenant boundaries. These tests attempt cross-tenant data access, unscoped queries, and privilege escalation. If any test fails, the build is rejected.
  • Chaos Engineering Fault Injection: We deliberately inject failures into the isolation layers to verify that the remaining layers hold. What happens when a network policy is misconfigured? What happens when a container escapes its namespace? What happens when a key rotation fails midway? We test these scenarios continuously.
  • Security-Focused Code Review: Every pull request that touches tenant-scoped logic requires review from a security engineer. This is not optional. The review checklist explicitly covers tenant boundary enforcement, credential handling, and query scoping.

The Bottom Line

Tenant isolation in a BYODb architecture is not a single feature. It is a system of interlocking controls that span identity, application logic, data storage, encryption, networking, and compute. Each layer is designed to hold independently, so a failure in one does not cascade into a breach.
